Variable Length Pointers

A fixed-width memory bus can only address so many bytes.

n bit bus 2^n bytes addressable
8 256
16 64KiB
32 4GiB
64 16EiB

These are like wchar_t code units.

One could perhaps address more by having a memory management unit that uses UTF-8000-style pointers. One could already achieve this by eg having a 16 bit processor just use 32 bit pointers

I think often about Ben Eater's 8-bit breadboard computer.

Verdict

Difficult to implement due to the variable length that needs reserving in a block of memory. UTF-8000 is best for encoding data for storage and transmission, not using within a program.

DNS or ICANN Style Private Codepoint Allocation

There are currently three Private Use Areas of codepoints allocated by the Unicode Consortium. Those are U+E000 to U+F8FF in plane 0, almost all of plane 15, and almost all of plane 16, where the only non-private codepoints in planes 15 and 16 are U+FFFFE, U+FFFFF, U+10FFFE and U+10FFFF. This is a total of 137,468 out of 1,114,112 (12%) codepoints, and is 137,468 out of 299,448 (46%) allocated codepoints. This may seem a lot, but if everyone on the planet wanted their own block of codepoints, like IP addresses, we would soon run out.

Perhaps a large range way beyond the current U+10FFFF upper limit could be allocated as private use. Authority over this private range could be given to an organization like ICANN, who could rent them out like IP addresses or domain names. E.g. for 26 byte (131 content bits) UTF-8000, 128-bit codepoints of the form U+7_XXXX_XXXX_XXXX_XXXX_XXXX_XXXX_XXXX_XXXX could be made publicly available.

This would allow people to put their own symbols there, logos for companies, personal signatures etc, with fonts supplied via a DNS-like lookup service. Unicode has already said that they will decline logos, brands, paintings etc as codepoints, but perhaps a 128 bit zone like U+7_XXXX_..._XXXX above could be allocated for fun. Expiration of codepoints would be the greatest issue for this idea. Unicode makes its codepoints permanent and never deletes them so as to not break compatibility. This idea is a bit too corporate.
