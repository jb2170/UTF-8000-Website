Properties of UTF-8000

Preserved from UTF-8

ASCII (1 byte UTF-8000) has 7 bits of content and 0 mandatory content bits.

2-byte UTF-8 (2 byte UTF-8000) has 11 bits of content (which is not a special case), but only 4 mandatory content bits.

Glossary / Byte Anatomy / Implications
Term Definition

Code Unit

A collection of UTF-8000 bytes that encode a single codepoint.

First Byte

The first, one and only, byte that begins a UTF-8000 Code Unit. We can tell that a byte is a First Byte because it does not begin with the Continuation Byte's characterizing prefix 10. This is part of what makes UTF-8 and UTF-8000 Self Synchronizing; we know where a Code Unit starts.

This term is also known as First Start Byte.

This term is mutually exclusive with Continuation Byte.

This term is not synonymous with Start Byte.

Continuation Byte

A byte beyond the First Byte of a UTF-8000 Code Unit. It is characterized by its highest bits being 10 giving it the form 10yyyyyy. The y bits may be Start Bits, Content Bits, or Mandatory Content Bits depending on the context of where that Continuation Byte is in its Code Unit.

This term is mutually exclusive with First Byte.

The continuation prefix bits are highlighted in bright cyan.

Start Byte

A byte containing one or more Start Bits. Start Bytes exist contiguously at the beginning of a UTF-8000 Code Unit. The power of UTF-8000 is that we can have many Start Bytes, to achieve arbitrary Code Unit lengths, to encode arbitrarily large Codepoints.

Every Code Unit has at least one start byte. The first Start Byte is the First Byte, and is followed by zero or more Continuation Bytes that are Start Bytes. Because a UTF-8000 Code Unit can have multiple Start Bytes, this term is not synonymous with First Byte.

In restricting to only UTF-8 without UTF-8000, this term is synonymous with First Byte. This is because UTF-8 Code Units only require one Start Byte, whether using up to 4 bytes in the current UTF-8 standard (RFC 3629 (2003)), or using up to 6 bytes in former standards (RFC 2044 (1996) and RFC 2279 (1998)).

Start Bits

The block of Y Bits at the beginning of a UTF-8000 Code Unit that are all 1 bits, and the first 0 bit that terminates these ones. To be clear, the Start Bits includes this terminating zero bit.

For an ASCII Code Unit the Start Bits sequence is 0, a single zero bit in the highest bit of the First Byte. This characterizes ASCII bytes into the form 0xxxxxxx.

For a non-ASCII Code Unit, made of n bytes, the Start Bits are n 1 bits followed by a terminating 0 bit. Thus the Start Bits sequence is of length n+1 and is of the form 111...10.

The Start Bits are part of what makes UTF-8 and UTF-8000 Self Synchronizing; we know how many bytes we expect in a Code Unit.

This term is mutually exclusive with Content Bits.

These bits are highlighted in bright magenta.

Content Byte

A byte containing one or more Content Bits.

A byte being a Content Byte does not imply that it is a Continuation Byte, as for example a 3-byte Code Unit begins with 1110xxxx, which is not a Continuation Byte, and contains 4 Content Bits.

A byte being a Continuation Byte does not imply that it is a Content Byte, as for example a 22-byte Code Unit contains 10111111 as its second byte.

Content Bits

The block of Y bits in a Code Unit beyond the Start Bits, in which the Codepoint's binary bits are stored. For example a 2-byte Code Unit, which has the form 110xxxxx 10xxxxxx, has 11 Content Bits.

This term is mutually exclusive with Start Bits.

These bits are highlighted in lime, or bright lime if they are Mandatory Content Bits.

Mandatory Content Byte

A byte containing one or more Mandatory Content Bits.

Mandatory Content Bits

The first zero, four, or five Content Bits of a Code Unit in which there must be at least one 1 bit, lest the bytes be an Overlong Encoding, which is forbidden.

For ASCII there are none.

For 2-byte UTF-8000 there are four.

Naming

UTF-8000

Inspired by Python 3's development codenames in PEP 3000.

Pros

The thousand in UTF eight thousand sounds big and futuristic. This encoding scheme should also last for a thousand years plus!

Cons

The zeros are repetitive.

Having to remember exactly three zeros to make 8000. Some people may read it as eight hundred, or eighty thousand etc.

The inspiration logic doesn't exactly match up with Python because Python was moving from Python 2 to Python 3, not Python 3 to Python 3000, whereas we're going from UTF-8 to UTF-8000.

UTF-8K

Inspired by another of Python 3's development codenames Py3K / Py3k, we could use UTF-8K, the K capitalized like the UTF. It's also shorter than 8000.

UTF-8 and Knuckles

The K in UTF-8K reminds me of Sonic 3 & Knuckles, sometimes abbreviated to S3K. Sonic 3 & Knuckles is a XXX of Sonic 3 and Sonic & Knuckles using lock-on technology. In a similar way UTF-8 extends from ASCII, and UTF-8000 continues this extension.

Pros

Knuckles is cool.

Cons

SEGA might not be happy, though they do seem nicer than Nintendo with respect to fanart.

The hardest work of extending from ASCII has already been achieved by UTF-8. Is UTF-8000 lock-on? Not really.

Logo

UTF-8 does not have a logo. Therefore I had a bit of fun designing a logo for UTF-8000.

It's an eagle with UTF written across its wings and chest. The roman numerals for 8, VIII, flank its head. Below its claws it holds a fasces, wrapped with a UTF-8000 continuation byte of the form 10XXXXXX. This represents the unified strength of a bundle of UTF-8000 continuation bytes, which makes us unstoppable in conquering the entire integers, in the name of annexing them into Unicode codepoints, encoded as UTF-8000 bytes. The central fascis which sticks out perhaps represents the first start byte of a UTF-8000 sequence. This byte looks different from its continuation bytes, and has the power to start a sequence, which is represented by the wielding of the axe.

The favicon for this website is just the roman numerals VIII.
